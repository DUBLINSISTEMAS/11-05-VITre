## Conselho-5-agentes — decisões importantes que vieram dele

1. **C.2 antes de 1.4** — lista de produto vira o "esqueleto" reusado por categorias/banners/pedidos. Refatorar 4 listas depois custaria mais.
2. **Offset > cursor pagination** — Sandra tem ~50 produtos, não 50k. Index em `(store_id, created_at DESC)` cobre offset até 10k linhas. Cursor seria over-engineering.
3. **Filtros via `searchParams` (RSC nativo)** — não recriar `useState` no client. URL compartilhável + `revalidateTag` funciona + RSC re-renderiza sem complicação.
4. **Tree de categorias = adjacency list 2 níveis** — schema já fixou `parent_id` self-FK. Sem CTE recursivo, sem materialized path. Duas queries (raízes + filhas) e monta árvore em JS.
5. **Promoção segue inline no `productTable`** — não criar tabela separada agora. Sandra não pediu desconto por categoria/global. Helper `getEffectivePrice` evita drift entre lista/detalhe/carrinho/snapshot.
6. **Lista de pedidos pode ser feita junto** — UI cheap; só vai ter dado real depois da Fase 1.5 (storefront + checkout). Empty state precisa ficar bom.

## Aprendizados

- **Quando o conselho diz "GO COM RESSALVAS", as ressalvas são onde o valor real está.** O "antes de C.2 extraia `getEffectivePrice()`" salvou de ter promo logic espalhada em 5 lugares; agora cabe em 1 helper.

- **`router.refresh()` + `revalidatePath` server é suficiente** pra UX live em mutações — não precisei de optimistic UI em nenhuma das 5 listas (categorias, banners, pedidos, etc). Padrão simples e sem bugs de revert.
